### 💡 为什么选择 Docker?
我们理解并不是每个人都喜欢使用 Docker；但此方法是我们项目设计和运行效率的核心。我们将项目对 Docker 的承诺视为根本性的一部分，并鼓励那些希望寻找其他部署方法的人可以考虑社区驱动的替代方案。


#### **问：如何自定义标志和品牌？**

**答：** 您可以通过我们的 **[企业许可证](https://docs.openwebui.com/enterprise)** 自定义主题、标志和品牌，该许可解锁了专属的企业功能。

关于企业解决方案和品牌定制的更多详情，请联系销售团队：📧 **sales@openwebui.com**

#### **问：为什么要求我注册？我的数据会发送到哪里？**

**答：** 我们要求您注册成为管理员用户，以增强安全性。这可以确保即使 Open WebUI 暴露在外部访问中，您的数据仍然安全。需要注意的是，所有内容都保持在本地。我们不会收集您的数据。注册时，所有信息都留在您的服务器内，仅保存在您的设备上。您的隐私和安全是我们的首要任务，确保您的数据始终掌握在您手中。

#### **问：为什么我的 Docker 容器无法使用 `localhost` 连接到主机上的服务？**

**答：** 在 Docker 容器中，`localhost` 指的是容器自身，而不是主机。这种区别对网络非常重要。要从容器连接到主机上运行的服务，应该使用 DNS 名称 `host.docker.internal` 而不是 `localhost`。此 DNS 名称是 Docker 特别识别的名称，用于简化这种连接，将主机视为容器中可访问的实体，从而绕过通常的 `localhost` 范围限制。

#### **问：如何让主机上的服务对 Docker 容器可访问？**

**答：** 要使主机上运行的服务可供 Docker 容器访问，请将这些服务配置为监听所有网络接口，并使用 IP 地址 `0.0.0.0`，而不是仅限于 `127.0.0.1` 的 `localhost`。此配置可以使服务接受来自任何 IP 地址的连接，包括 Docker 容器连接。需要注意这种设置的安全影响，特别是在可能有外部访问的环境中。通过实施适当的安全措施，如防火墙和认证，可以降低风险。

#### **问：为什么我的 Open WebUI 没有更新？我重新拉取/重启容器后没有任何变化。**

**答：** 更新 Open WebUI 不仅仅是拉取新的 Docker 镜像。以下是为什么您的更新可能没有显示以及如何确保更新成功：

1. **更新 Docker 镜像：** 命令 `docker pull ghcr.io/open-webui/open-webui:main` 更新了 Docker 镜像，但并未更新运行中的容器或其数据。
2. **Docker 卷中的持久化数据：** Docker 卷独立于容器生命周期存储数据，确保您的数据（如聊天历史记录）在更新中保留。
3. **应用更新：** 确保通过移除现有容器（不会删除卷）并使用更新的镜像和现有的卷创建一个新容器来生效更新。

这一过程在保持数据安全的同时更新了应用程序。

#### **问：等等，为什么我要删除我的容器？我不会丢失我的数据吗？**

**答：** 这是一个常见的担忧，但如果正确使用 Docker 卷，删除容器并不会导致数据丢失。原因如下：

- **卷保留数据：** Docker 卷的设计目的是存储数据，独立于容器的生命周期。只要您的数据存储在卷中，无论容器发生什么，它都会保持完好无损。
- **安全更新过程：** 在更新 Open WebUI 时，删除旧容器并使用更新的镜像创建新容器不会影响存储在卷中的数据。关键是不要使用像 `docker volume rm` 这样的命令显式删除卷。

通过正确的更新步骤——拉取新镜像，移除旧容器（不删除卷），然后使用更新的镜像和现有卷创建新容器——您的应用程序代码会更新，同时数据保持不变且安全。

#### **问：我应该使用发行版打包的 Docker 还是官方 Docker 包？**

**A:** 我们建议使用官方的 Docker 包来运行 Open WebUI，而不是使用发行版打包的版本。官方的 Docker 包经常更新最新功能、错误修复和安全补丁，以确保最佳性能和安全性。此外，它支持 `host.docker.internal` 等重要功能，这些功能可能在发行版打包版本中不可用。这一功能对于 Docker 容器中的网络配置和连接至关重要。

选择官方的 Docker 包，可以在不同环境中获得一致的行为，更可靠的故障排除支持，以及访问最新的 Docker 创新功能。更广泛的 Docker 社区和资源也更倾向于与官方包保持一致，为您提供大量信息和支持，帮助解决可能遇到的问题。

运行 Open WebUI 所需的一切，包括您的数据，都会保持在您的控制范围和服务器环境内，体现我们对隐私和安全的承诺。关于安装官方 Docker 包的指南，请参考 Docker 官方文档网站上的 [安装 Docker 引擎](https://docs.docker.com/engine/install/) 指南。

#### **Q: Docker 是否支持 GPU 加速？**

**A:** Docker 的 GPU 加速支持因平台而异。官方支持 GPU 加速的 Docker 包括 Windows 的 Docker 和 Linux 的 Docker 引擎。其他平台，如 Linux 的 Docker Desktop 和 MacOS，目前尚未提供 GPU 支持。这一限制对于需要 GPU 加速的应用程序来说非常重要。为获得最佳体验并使用 GPU 功能，我们建议在官方支持 GPU 集成的平台上使用 Docker。

#### **Q: 为什么 Open WebUI 强调使用 Docker?**

**A:** 使用 Docker 的决定源于其能够确保一致性、隔离依赖关系并简化跨不同环境的部署。Docker 最大程度减少了兼容性问题，并简化了 WebUI 的启动运行过程，无论底层系统是什么。这是项目维护者从战略角度出发做出的选择，利用 Docker 的这些优势，尽管 Docker 有一定的学习曲线，但从部署和维护的角度来看，其优势显著。我们理解 Docker 可能不是每个人的首选；然而，这种方法是我们项目设计和运行效率的核心部分。我们认为项目对 Docker 的承诺是一个基本方面，并鼓励那些寻求其他部署方法的人探索社区推动的替代方案。

#### **Q: 为什么我的部署中语音转文本（STT）和文本转语音（TTS）功能无法正常工作？**

**A:** 部署中的语音转文本（STT）和文本转语音（TTS）服务可能需要使用 HTTPS 才能正常工作。现代浏览器施加了一些安全措施，限制某些功能（包括 STT 和 TTS）只能在安全的 HTTPS 连接下运行。如果您的部署未配置使用 HTTPS，这些服务可能无法按预期运行。确保您的部署可以通过 HTTPS 访问可以解决这些问题，从而启用 STT/TTS 功能的全部功能。

#### **Q: 为什么 Open WebUI 没有内置 HTTPS 支持？**

**A:** 虽然我们理解用户希望项目包括内置 HTTPS 支持的一体化解决方案，但我们认为这样的做法无法充分满足用户群体的多样化需求。在项目中直接实现 HTTPS 可能会限制灵活性，并不一定符合所有用户的具体需求或偏好。为了确保每个人都能根据自己的特定环境定制设置，我们将 HTTPS 分发的实现留给用户进行生产部署。这一决定带来了更大的适应性和定制性。虽然我们没有提供关于设置 HTTPS 的官方文档，但有需求的用户可以向社区成员请求指导，社区成员或许可以根据他们的经验分享见解和建议。

#### **Q: 我更新、重启或安装了一些新的软件，现在 Open WebUI 无法正常工作了！**

**A:** 如果您的 Open WebUI 在更新或安装新软件后无法启动，这很可能与直接安装方法有关，尤其是在后端依赖项没有使用虚拟环境的情况下。直接安装可能对系统环境的更改（如更新或新安装改变现有依赖项）较为敏感。为避免冲突并确保稳定性，我们建议使用虚拟环境来管理您的后端的 `requirements.txt` 依赖项。这样可以将您的 Open WebUI 依赖项与系统包进行隔离，从而最大程度减少此类问题的风险。

#### **Q: 我更新/重启后我的登录无法正常工作了，我不得不创建一个新账户，并且所有聊天记录都消失了。**

**A:** 此问题通常出现在创建Docker容器时未为`/app/backend/data`挂载卷，或不小心删除了指定的Open WebUI卷（通常在我们示例中命名为`open-webui`）。Docker卷对于在容器生命周期中持久化数据非常重要。如果您在重启后需要创建新账户，这可能是由于您启动了一个新容器并未附加现有的数据卷。请确保Docker运行命令包含指向正确数据位置的卷挂载，以防止数据丢失。

#### **问：登录时失败了，我创建了一个新账户，现在被告知我的账户需要管理员激活。为什么？**

**答：**这种情况发生在您忘记了首次设置时创建的初始管理员账户的密码。第一个账户会自动指定为管理员账户。如果在没有访问管理员账户的情况下创建新账户，则需要管理员进行激活。避免丢失初始管理员账户凭据对于无缝访问和管理Open WebUI至关重要。请参考[重置管理员密码](troubleshooting/password-reset)指南了解如何恢复管理员账户。

#### **问：为什么Open WebUI无法启动并显示SSL错误？**

**答：**您在启动Open WebUI时遇到的SSL错误可能是由于缺少SSL证书或[huggingface.co](https://huggingface.co/)配置不正确造成的。为了解决此问题，您可以为HuggingFace设置一个镜像站，例如[hf-mirror.com](https://hf-mirror.com/)，并在启动Docker容器时将其指定为端点。在Docker运行命令中使用`-e HF_ENDPOINT=https://hf-mirror.com/`参数来定义HuggingFace镜像地址。例如，您可以如下修改Docker运行命令：

```bash
docker run -d -p 3000:8080 -e HF_ENDPOINT=https://hf-mirror.com/ --add-host=host.docker.internal:host-gateway -v open-webui:/app/backend/data --name open-webui --restart always ghcr.io/open-webui/open-webui:main
```
#### **问：Open WebUI中的RAG表现很差或者完全不起作用。为什么？**

**答：**如果您使用的是**Ollama**，请注意Ollama默认将上下文长度设置为**2048个token**。这意味着可能没有检索到的数据会被使用，因为它们无法适应可用的上下文窗口。

为了改善在Open WebUI中使用检索增强生成（**RAG**）的性能，您应该将上下文长度**增加到更大的值（例如8192个token及以上）**，以确保检索到的文档能够有效地为模型响应做出贡献。

为此，请配置您的**Ollama模型参数**以允许更大的上下文窗口。您可以在聊天界面或模型编辑页面直接检查并修改此设置，以显著增强RAG体验。

#### **问：Open WebUI是否支持MCP（模型上下文协议）？**
**答：**[是的，Open WebUI官方支持MCP工具服务器——但仅通过**兼容OpenAPI的代理**](/openapi-servers/mcp)（[openapi-servers](https://github.com/open-webui/openapi-servers)）来确保最佳的兼容性、安全性和可维护性。

为了桥接MCP（以及其他后端协议），我们提供了专门构建的代理实现，可在以下地址获取：👉 [https://github.com/open-webui/mcpo](https://github.com/open-webui/mcpo)

此设计选择基于以下几个核心原则：

- 📘 标准优先：OpenAPI是RESTful服务定义以及基于合同的服务互操作的事实标准。通过通过OpenAPI对集成进行对齐，我们可以在升级和部署中实现可复制和基于模式的行为。
- 🔒 安全模型隔离：通过代理集成MCP可以对后端协议行为进行沙盒化和隔离，从而减少攻击面并提供边界级别审核、身份验证和可观察性。
- ⚙️ 协议抽象：通过统一的OpenAPI模式支持异构后端（例如MCP），使得Open WebUI在保持可预测运行行为的同时，保持后端无关性。
- ⛓️ 解耦的部署拓扑：基于代理的体系结构允许工具服务器（例如MCP）独立于前端展示进行发展，支持高度模块化的生产环境或分布式工作负载。

综上所述：支持MCP——只要MCP工具服务器通过兼容OpenAPI的代理前置。此架构决策是经过深思熟虑的，旨在确保Open WebUI具有可扩展性、安全性和可维护性。
